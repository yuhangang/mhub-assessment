Workflow State Engine Lifecycle

This flowchart describes the life cycle of a transaction-wrapped workflow trigger and its approval stages.

01
1. Event Triggered
An action is taken on a domain entity (e.g. Booking cancellation requested). The engine receives the trigger name and initiator ID.
event: 'booking.cancellation_requested'
entity: 'booking' | id: '501'
02
2. Template Resolution
Checks if the trigger event is enabled and resolves the active workflow template blueprint. Ensures no active instance already exists.
workflow_templates WHERE trigger_event = ? AND is_active = 1
03
3. Database Transaction
Runs in a secure DB transaction: creates the instance, clones all template steps, and sets Step 1 to AWAITING_ACTION.
db.transaction(() => {
  INSERT IN instances;
  INSERT IN instance_steps;
})
04
4. Auth & Step Action
Assignee (Agent matching user ID or role) reviews the active step. Performs step authorization checks before actioning.
Check: user.role === assignee_role OR user.id === assignee_user_id
05
5. Concurrency Check
Applies Optimistic Locking via a version field. Prevents two users from approving/rejecting the same step simultaneously.
UPDATE instance_steps SET status = ?, version = version + 1 WHERE id = ? AND version = ?
06
6a. Action Approved
Step is marked as approved, and an audit trail decision is written. The engine queries for the next step in sequence sequence.
SELECT WHERE sequence > current_seq
IF EXISTS: status = 'awaiting_action'
IF NONE: instance.status = 'approved'
07
6b. Action Rejected
Rejection immediately fails the workflow. The step status is marked as rejected, a rejection comment is required, and the instance fails.
instance.status = 'rejected'
Enforce comment length > 0
08
7. Completion Callback
If the last step is approved, the instance is approved and the engine triggers the registered hook to finalize domain operations.
callbackFn(entityId)
Booking status $\rightarrow$ 'cancelled'
Unit status $\rightarrow$ 'available'

Concurrency Control & Optimistic Locking

In a high-concurrency property booking engine, two roles might open the review interface and try to click "Approve" at the exact same moment. If they do, a classic race condition occurs where both attempts succeed, resulting in multiple step audit logs and corrupted state.

To solve this safely without heavy database table locking, we implement Optimistic Locking via a version column on the workflow_instance_steps table.

How it works:

  1. When the user reads the step data to render it on screen, the system fetches the current step record, including its version (e.g. version = 0).
  2. When they submit an action (approve/reject), the system issues an update statement targeting that specific version:
UPDATE workflow_instance_steps SET status = 'approved', version = version + 1, updated_at = CURRENT_TIMESTAMP WHERE id = ? AND status = 'awaiting_action' AND version = ?;

If another user successfully updated the row a microsecond earlier, the version in the database would already be 1. The query's WHERE version = 0 clause will fail to match any rows, and SQLite will report that 0 changes were made. The engine detects this conflict, rolls back the transaction, and returns a 409 Concurrency Conflict error.

Polymorphic Entity Referencing

Workflows shouldn't just be built for "Bookings". Tomorrow, the business might require approval workflows for "Projects" (e.g. launching a project), "Discount Vouchers", or "Agents Profiles".

To make the workflow engine generic and decoupled from specific business tables, we implement a Polymorphic Relationship on the workflow_instances table using two columns:

  • entity_type (TEXT): String identifying the target domain entity (e.g. 'booking', 'project').
  • entity_id (TEXT): The primary key identifier of the targeted row in the target table (stored as text to support UUIDs or string IDs).

Design Note: Since SQLite does not support native polymorphic foreign keys directly, referential integrity and polymorphic checks are validated inside the application service layer (in WorkflowEngineService) before initiation, ensuring target entities exist.

One Active Workflow Rule

To prevent an entity from undergoing multiple concurrent workflows (e.g. attempting to request cancellation twice), we enforce a unique partial index on active instances:

CREATE UNIQUE INDEX IF NOT EXISTS idx_active_entity_instance ON workflow_instances(entity_type, entity_id) WHERE status IN ('pending', 'in_progress');

Referential Integrity & Enforcements

A major vulnerability in workflow systems is step hijacking or orphaned decision logs. The schema includes rigid relational safeguards:

1. Composite Foreign Keys

When writing a decision audit log to the workflow_step_decisions table, how do we guarantee that the step_id belongs to the referenced instance_id?

By using a composite foreign key referencing multiple columns:

-- In workflow_step_decisions creation FOREIGN KEY (step_id, instance_id) REFERENCES workflow_instance_steps(id, instance_id) ON DELETE CASCADE

This prevents an attacker from sending a POST request that updates a step belonging to Instance A while assigning the decision audit log to Instance B.

2. Enforcing One Awaiting Step

Workflows must run sequentially. We should never have both Step 1 and Step 2 awaiting action at the same time. To enforce this database-wide, we define a partial unique index:

CREATE UNIQUE INDEX IF NOT EXISTS idx_one_awaiting_step_per_instance ON workflow_instance_steps(instance_id) WHERE status = 'awaiting_action';

This ensures that SQLite will reject any transaction that attempts to set more than one step to awaiting_action within the same instance.

Part 3 Code Review Improvements

During our assessment review, we refactored the POST /api/review/workflow-instances/:id/steps/:stepId/approve route to solve critical system design issues. Here are the core issues resolved:

  • SQL Injection: Changed direct string interpolation to clean parameterized placeholder queries (?).
  • Inverted Status Logic: Fixed a bug where actions were processed even if step status was already actioned. Added guard clauses rejecting non-pending actions.
  • Race Conditions: Introduced Optimistic Locking to block concurrent double-approvals, returning a 409 Conflict.
  • Authorization Check: Validated that the actioning user matches either the assignee's user ID or has the required role (e.g. Sales Manager, Finance Manager).
  • No Transaction Wrapping: Grouped all state changes (updating step status, writing decision audit trail, and updating next steps) inside a transaction block (db.transaction()) to guarantee atomicity.
